# CLAUDE.md

Guidance for working in this repository.

## What this is

**Laufbahn** (German: "career path") ‚Äî a personal job-application tracker. Track every
application from *discovered* to *offer* on a Kanban pipeline, with interviews, contacts,
follow-ups, an activity timeline, a dashboard, a user profile, AI-generated cover letters,
and a local agentic LLM assistant that never sends data to a third-party AI provider.

Hosting is **hybrid**: the app always runs locally; real dev data lives in a managed
cloud Postgres (Neon, via `DATABASE_URL` in `backend/.env`). Demo + test DBs stay in
local Docker.

It is a portfolio piece: prioritize clean, layered, well-typed code over cleverness.

## Tech stack

**Backend** (`backend/`)
- Node 22, Fastify 5, TypeScript (strict, ESM ‚Äî imports use `.js` extensions).
- PostgreSQL 17 via Drizzle ORM (`postgres-js` driver) ‚Äî cloud-hosted on Neon for dev
  data; Dockerized for demo/tests. Migrations are **generated**, not hand-written.
- Zod for validation *and* response serialization (`fastify-type-provider-zod`).
- JWT auth (access + refresh) in httpOnly cookies; bcryptjs for hashing.
- Vitest integration tests run against a **real** Postgres (no DB mocks).
- Local LLM via Ollama (OpenAI-compatible `/v1` endpoint). The assistant + cover-letter
  generation both call `modules/assistant/assistant.llm.ts#chatCompletion`.

**Frontend** (`frontend/`)
- React 19, Vite, TypeScript (strict).
- TanStack Router (file-based, `src/routes/`) + TanStack Query for server state.
- Jotai for local UI state; React Hook Form + Zod for forms.
- Tailwind CSS v4, shadcn-style primitives in `src/components/ui/`.

## Architecture & conventions

**Backend layering ‚Äî every feature is a module under `backend/src/modules/<name>/`:**

```
routes ‚Üí controller ‚Üí service ‚Üí repository ‚Üí database
```

- `*.routes.ts` ‚Äî wires Zod schemas + dependency injection; registers under a prefix in `app.ts`.
- `*.controller.ts` ‚Äî thin; calls `requireUser(request)`, delegates to the service.
- `*.service.ts` ‚Äî business rules + row‚ÜíDTO mapping. Throws typed errors from `lib/errors.ts`.
- `*.repository.ts` ‚Äî Drizzle queries. **Every query is scoped to `userId`** (per-user data isolation).
- `*.schemas.ts` ‚Äî Zod schemas + inferred types (request bodies, query, DTOs).
- `*.test.ts` ‚Äî Vitest integration tests via `app.inject`.

When adding a backend feature, copy the shape of an existing module (`applications` is the
fullest example; `profile` is the simplest). Register its routes in `backend/src/app.ts`.
Add a typed error to `lib/errors.ts` rather than throwing strings. Mirror new DTOs into
`frontend/src/lib/types.ts`.

**Frontend ‚Äî feature-sliced under `src/features/<name>/`:**
- `api.ts` ‚Äî TanStack Query hooks using the shared `@/lib/api` helper (`get/post/put/patch/delete`).
- Components co-located in the feature folder; routes in `src/routes/` (auth-gated routes live
  under `_authed/`). The router tree (`routeTree.gen.ts`) is auto-generated by the Vite plugin.

**Conventions**
- No self-explaining comments. Comment *why*, not *what*.
- Match the style of surrounding code (naming, error handling, DI wiring).
- Keep data scoped per user; never leak another user's rows (return 404 on cross-user access).

## Key features / modules

- `auth`, `applications` (status pipeline ‚Äî no stage skipping), `interviews`, `contacts`,
  `follow-ups`, `activities` (auto-logged timeline), `dashboard` (stats).
- `assistant` ‚Äî agentic tool loop over the user's data; write actions are *proposed* and
  require explicit confirmation via `/api/assistant/execute`.
- `profile` ‚Äî one profile per user (`PUT /api/profile` upserts; `GET` returns `null` if unset).
  Structured fields + a freeform markdown CV summary. Powers cover-letter generation.
- `cover-letters` ‚Äî generate a letter from profile + a job (pasted text, a fetched URL, or a
  linked application) + optional format references. Letters are persisted and optionally linked
  to an application (`ON DELETE SET NULL`). `lib/job-fetch.ts` fetches job URLs with an **SSRF
  guard** (blocks private/loopback/link-local hosts) ‚Äî treat that guard as security-critical.

## Running the project ‚Äî IMPORTANT

There are **two run modes; never run both at once** (they both bind port 3000).
All commands are in the root `Taskfile.yml` ([go-task](https://taskfile.dev); `task --list`).

### üÖ∞ Demo mode ‚Äî everything in Docker (recruiter-ready, self-contained local DB)
```bash
task demo      # build images + start all + seed demo data
task down      # stop (keeps DB) ¬∑  task up  # start ¬∑  task rebuild  # after new code
```
- Frontend http://localhost:5173 ¬∑ API http://localhost:3000
- Demo login: **recruiter@laufbahn.app** / **laufbahn-demo**

### üÖ± Dev mode ‚Äî app local with hot reload, real data in the cloud DB (Neon)
```bash
task dev:migrate   # apply migrations to the cloud DB (DATABASE_URL in backend/.env)
task dev:api       # terminal 1 ‚Äî API hot reload (:3000)
task dev:web       # terminal 2 ‚Äî frontend hot reload (:5173)
```
- Offline fallback: `task dev:local-db` starts a local Postgres on :5432; flip the
  commented-out `DATABASE_URL` in `backend/.env`. `task db:push-to-cloud` copies the
  local DB up to the cloud one (destructive on the cloud side).

### Other
```bash
task seed      # (re)load demo data ‚Äî idempotent (deletes & recreates the demo user)
task test      # backend integration tests
task reset     # destroy the demo DB volume, start fresh + migrate (never the cloud DB)
task --list    # list all tasks
```

## Environment & config ‚Äî gotchas

- **`backend/.env`** is read by **Dev mode** (`pnpm dev` / `task dev:api`): `DATABASE_URL`
  points at the **Neon cloud DB** (use the *direct*, non `-pooler` connection string ‚Äî
  `db/client.ts` disables prepared statements automatically if a pooled URL is used).
- **Root `.env`** is read by **docker compose** (Demo mode). `cp .env.example .env` first time.
- Three databases exist:
  - **dev DB** ‚Üí Neon (cloud) ‚Äî your real data. Migrations run via `task dev:migrate`.
  - demo DB ‚Üí `postgres` container (host port **5432**) ‚Äî demo-mode data + offline fallback.
  - **test DB** ‚Üí `postgres_test` container (host port **5433**) ‚Äî memory-backed, used
    **only** by `task test`, wiped on restart. Not real data; ignore it for normal use.
- Tests use their own DATABASE_URL (see `backend/vitest.config.ts`), independent of `.env`.

## Database changes

1. Edit `backend/src/db/schema.ts`.
2. `pnpm --filter backend db:generate` ‚Üí emits a migration in `backend/src/db/migrations/`. Commit it.
3. Migrations apply automatically on container boot (Docker entrypoint) and via
   `pnpm --filter backend db:migrate` locally.
4. If a new table holds per-user data, add it to the `TABLES` truncation list in
   `backend/test/setup.ts`, and extend the seed in `backend/src/db/seed.ts`.

## Spec-driven changes (OpenSpec)

Larger features are proposed under `openspec/changes/<name>/` (proposal, design, specs, tasks)
before implementation. The most recent example: `add-user-profile-cover-letters`. Follow that
structure for substantial new capabilities.

## Before you call something done

- Backend: `pnpm --filter backend test` (needs the test DB: `task test` handles it) and
  `pnpm --filter backend typecheck`.
- Frontend: `pnpm --filter frontend typecheck` (and `vite build` regenerates the route tree).
- Report failures honestly with their output; don't mark work complete unless it's verified.
